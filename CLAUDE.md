# CLAUDE.md

## Project Overview

Galaxy Map — minimal task tracker / kanban board for AI agents. REST API + React UI + MCP server.

## Documentation

**Project-level docs** (proposals, features, design, integration guides) live in a separate documentation repo:
- **Location**: `~/git/obsidian/projects/galaxy-map/` (on local workstation)
- **Structure**: Organized by domain (`features/`, `architecture/`, `integration/`)
- **Sync**: Features have both a doc in that repo AND a task in Galaxy Map (with cross-references)

When working on a feature:
1. Check the doc in `~/git/obsidian/projects/galaxy-map/features/[feature]/` for context
2. Find the corresponding Galaxy Map task (linked in the doc)
3. Update both as you work

## Architecture

- **Backend**: Python FastAPI + async SQLite (`backend/main.py`), database file is `board.db`
- **Config**: YAML-based status configuration (`backend/config/statuses.yaml`) loaded on startup
- **Frontend**: React 18 + Vite (`frontend/src/App.jsx`), fetches statuses from API on mount, polls backend every 3 seconds
- No auth, no websockets

## Quick Start

```bash

# Frontend
cd frontend && npm install && npm run dev
```

## Key Concepts

- **Statuses (columns)**: Dynamic, loaded from `config/statuses.yaml`. Each status has: key, label, description, order, color, allowed_transitions, terminal flag. Current statuses: backlog → queued → in_progress → needs_review → needs_human → done
- **Status Descriptions**: Agents can fetch full status metadata including descriptions via `/api/statuses` or `/api/statuses/{status_key}` to understand when to use each status
- **Specializations**: general, intake, coding, planning, research, claude-code
- **Task IDs**: server-generated by default, but clients can provide their own `id` on creation (409 if duplicate)
- **Dependencies**: tasks have `blocked_by` list with lazy validation — references to non-existent task IDs are accepted and treated as non-blocking. `is_blocked` is true only when a blocker exists AND isn't `done` or `needs_review`
- **Metadata**: arbitrary JSON field on tasks for agent context

## API

All endpoints under `/api`. Main ones:
- **Tasks**: `GET/POST /tasks`, `GET/PATCH/DELETE /tasks/{id}`, `POST /tasks/{id}/claim`
- **Statuses**: `GET /statuses` (list all with descriptions), `GET /statuses/{status_key}` (single status)
- **Specializations**: `GET /specializations`
- **Health**: `GET /health`

Status endpoints return full metadata including descriptions, allowing agents to make informed decisions about task status.

## Config (statuses.yaml)

Located at `backend/config/statuses.yaml`. Defines all valid statuses with metadata:

```yaml
statuses:
  - key: backlog
    label: Backlog
    description: Task is planned but not yet ready to start. Waiting for resources or dependencies.
    order: 0
    color: "#71717a"
    allowed_transitions: [queued]
    terminal: false
```

Fields:
- `key`: Identifier used in API and database
- `label`: Display name for UI
- `description`: Human-readable description for agents
- `order`: Column position (0-5)
- `color`: Hex color for column header
- `allowed_transitions`: Valid next statuses (informational, not enforced yet)
- `terminal`: True if status is final (no transitions allowed)

## MCP Server

- **Location**: `mcp-server/server.py` — wraps the REST API as MCP tools using `FastMCP`
- **Transport**: stdio (local) or SSE (Docker)
- **Config**: `GALAXY_MAP_URL` env var (default `http://localhost:8000`), `MCP_TRANSPORT` env var (default `stdio`)
- **Tools**: 
  - Task management: `list_tasks`, `create_task`, `get_task`, `update_task`, `delete_task`, `claim_task`, `merge_metadata`
  - Status info: `list_statuses` (with descriptions), `get_status_details` (single status with allowed transitions)
  - Other: `list_specializations`, `get_task_stats`
- **Install**: pip-installable package, exposes `galaxy-map-mcp` console script
- **Docker**: runs as `galaxy-map-mcp` service in docker-compose, SSE transport on port 8080

```bash
# Local
pip install ./mcp-server && galaxy-map-mcp

# Docker (included in docker-compose.yml)
docker compose up galaxy-map-mcp
```

### Using Status Information in Agents

Agents should:
1. Call `list_statuses()` to see all available statuses with descriptions
2. Call `get_status_details(status_key)` to check allowed transitions before updating
3. Use status descriptions to understand when to move a task

Example:
```python
# Get status info to make informed decision
status_info = get_status_details("in_progress")
print(status_info["description"])
# "Task is actively being worked on by an agent."
print(status_info["allowed_transitions"])
# ["needs_review", "needs_human", "backlog"]
```

## Dev Notes

- Backend has no tests yet
- Frontend is a single-file app (`App.jsx`) with multiple components: `App`, `TaskDetailModal`, `BlockerEditor`, `TaskCard`, `Column`, `AddTaskForm`
- Frontend dynamically fetches statuses from API and builds columns from the response (no hardcoding)
- Task detail modal opens on card click — supports inline editing of title, description, specialization, priority, and blockers
- SQLite DB is a single file — no migrations system
- Status configuration is loaded once on startup via lifespan hook
- Migration script available: `backend/migrate_tasks.py` (migrates tasks with invalid statuses to `needs_human`)
